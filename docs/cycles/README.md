# Cycles

Within each cycle directory is a collection of feature sub-folders.

Each feature folder contains:

- requirements.md
- specification.md

...used to drive AI-assisted code generation.

Each cycle also has a *session-log* directory containing files summarizing the work done, decisions made, and next steps. This allows for easy resumption in future sessions by providing a clear context snapshot.

## Workflow

```
requirements.md  →  specification.md  →  generated code
```

1. **Requirements** define *what* and *why* — user stories, acceptance criteria, scope
2. **Specifications** define *how* — data models, API contracts, component behaviour, edge cases, constraints
3. **Code** is generated by providing a spec file as context to the AI

## Specifications

### One Spec Per Feature

Each specification file corresponds to a single feature or coherent unit of work. Keeping specs focused ensures the AI has a tight, unambiguous context window.

### What a Good Spec Includes

- **Behaviour and constraints** — what the system does, not how it's implemented
- **Inputs and outputs** — API shapes, component props, function signatures
- **Edge cases** — what happens when data is missing, invalid, or at limits
- **Acceptance criteria** — these should be precise enough to serve as test cases
- **Explicit exclusions** — what this spec does *not* cover, to prevent scope creep

### What to Avoid

- Duplicating information already in the requirements file — link to it instead
- Implementation detail (e.g. which ORM method to call) — leave that to the AI
- Vague language — prefer "returns HTTP 404 if job not found" over "handles errors gracefully"

## Accuracy

Feature folder documents are correct at the time of the cycle.

If the feature changes in a different cycle, it will have a new folder created in that cycle,

Requirements and Specifications are only updated within the _current_ cycle - after that they become a static record.
